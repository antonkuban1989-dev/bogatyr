# CLAUDE.md — проект «Богатырь»

Технический контекст проекта. Claude Code читает этот файл при старте.
Финансовые данные и бизнес-цифры здесь намеренно отсутствуют — репозиторий публичный.

---

## Что это

**Богатырь** — собственная система управления двумя бизнесами владельца:
маркетплейсы (Wildberries, 4 ИП) и пилатес-студия.
Задачи → данные → отчёты → ИИ-помощник. Строится самостоятельно, чтобы понимать
домен и владеть активом, а не покупать готовое.

Имя: Илья Муромец — тридцать лет набирал силу, потом встал. Сначала фундамент, потом сила.

### Цикл обновления кода
1. Правка в мастерской → `git add . && git commit -m "..." && git push`
2. На сервере → `git pull` → **`systemctl restart bogatyr`** ← без рестарта работает старый код

### Команды
- `systemctl restart|stop|status bogatyr`
- `journalctl -u bogatyr -n 50` — логи при ошибке
- Ручной запуск: `cd ~/bogatyr && source venv/bin/activate && python zadachnik.py`
- Возврат в мастерскую: `wsl` → `cd ~/bogatyr` → `source venv/bin/activate`

### Различать окружения
Приглашение `anton@DESKTOP...` — мастерская: правка кода, git push.
Приглашение `root@...` — сервер: запуск, `.env`, установка, Claude Code.

## Сетевые ограничения (важно)

С российского IP заблокированы **и** Anthropic (403), **и** Telegram (Network unreachable).
Поэтому сервер в Казахстане: оттуда Telegram отвечает 302, Anthropic 404 — доступ открыт.
Локальный запуск бота с ноутбука невозможен.

Белые списки РФ (MAX, ВК, Яндекс) работают в одну сторону: дают доступ к российским
сервисам при отключениях, но не являются туннелем наружу.

Перспектива: второй VPS в РФ для российских API (гибридная схема РФ + Казахстан).

## Что построено

**Задачник** (`zadachnik.py`) — Flask-приложение:
- память в файле, переживает перезапуск
- задачи разовые (на дату) и повторяющиеся (по дням недели, с датой окончания)
- время по галочке, из списка с шагом 15 минут
- исполнители и категории с цветами — добавляются на лету, хранятся в `meta.json`
- фильтры: период (сегодня/неделя/месяц/год/произвольный) · исполнитель · категория
- сортировка: по дате / по времени (без времени — в конец) / по категории
- добавление и удаление со страницы
- **обратимое удаление цепочек**: одно вхождение (`cancelled`) или вся цепочка
  (`until` = вчера) — выполненное в прошлом не трогается
- вход по логину и паролю (HTTP Basic, перехватчик `before_request`)

Дефолты интерфейса: период — год, исполнители — все, категории — все, сортировка — по дате.
Задача по умолчанию **разовая**; галочка «Повторяющаяся» раскрывает дни недели и «до какой даты»
(пусто = 5 лет).

## Принципы (соблюдать)

1. **Один работающий этаж за заход.** Каждый шаг заканчивается тем, что работает и чем
   пользуются. Пустой шаблон не оживает. Не строить каркас «на будущее» — наполнять модуль
   за модулем реальными данными.
2. **Обратимость.** Всё создаваемое легко откатывается, но без потери истории: прошлое
   и выполненное неприкосновенно, управляемо только будущее. Напоминать об этом на каждом
   шаге, где появляется что-то создаваемое.
3. **Секреты только в `.env`.** Никогда в код, никогда в git, никогда в чат.
   `.gitignore` запрещает `.env`, `venv/`, `__pycache__/`.
4. **Не пилить то, что уже в коробке.** Покупать готовое там, где это не станет активом.
5. **90% автоматизации — обычный код по расписанию.** LLM звать точечно, только там, где
   нужно суждение. «Рой агентов» — антипаттерн, правильная схема — оркестратор.
6. **Мобильную адаптивность закладывать сразу**, не переделывать потом.
7. **К каждой системной задаче писать инструкцию** — она же становится готовым ТЗ
   для её последующей автоматизации.

## Архитектура (строим снизу вверх)

**Этаж 1 — Данные:** единый слой над WB API, YClients API, Дзен-мани, Google-таблицами.
Перспектива — собственный MCP-сервер.
**Этаж 2 — Память:** база с остатками, ценами, историей, baseline.
**Этаж 3 — Логика:** оркестратор по расписанию, без ИИ. Здесь 90% работы.
**Этаж 4 — Суждение:** LLM точечно (оценка новостей, ответы клиентам, тексты).
**Этаж 5 — Лицо:** веб-интерфейс, Telegram/MAX-бот, дашборды, потом голос.

**Инструменты сбоку:** Playwright/Firecrawl (парсинг конкурентов), Whisper (расшифровка),
Higgsfield/Kling (видео), Figma (доводка инфографики), рекламные API.

**Структура кабинета (цель):**
Кабинет → направления (Маркетплейсы, Фитнес) → подкатегории (ВБ, Озон; у Фитнеса нет)
→ модули (ДДС, остатки, баланс, отчёты, дашборды; у Фитнеса свои).
Блоки синхронизированы через общий слой данных.
**Логины и права — последний этаж, не первый.**

## План развития

**Задачник (ближайшее):**
- Вкладка **«Идеи»** — задачи без даты, чтобы не терялись
- Напоминания по желанию, канал на выбор (Telegram / MAX / WhatsApp), поля для телефона
  и адресов мессенджеров
- Приоритеты, подзадачи, статусы, комментарии
- Вложения — ТЗ к задаче, открывать прямо из задачи
- Контролёр задачи кроме исполнителя, лог выполнения
- Повтор «каждый N-й день/месяц/квартал»
- Памятки (дни рождения и подобное), не только задачи
- Мобильная адаптация, кастомные списки-чекбоксы

**Инфраструктура:**
- Claude Code на сервере (нужен Node ≥22)
- Домен вместо IP, HTTPS
- Второй VPS в РФ для российских API

**Маркетплейсы:**
- Автовыгрузка остатков и денег через WB API → Google-таблицы → дашборды `/wb`
  (страницы `/wb/investor`, `/wb/check-list`, `/wb/sklad` — шаблоны, привязать к данным)
- Вкладка «остатки»: приход/отгрузка/остаток вместо ручного пересчёта; приход берётся
  из вкладки себестоимость по дате доставки по РФ
- Форма недельного отчёта по номенклатуре
- **ИИ-разбор новостей** — не пересказ, а «что делать и что критично», с расчётом последствий
- Мобильное приложение с дашбордами, ДДС, балансом, триггерами *(большой блок)*

**Фитнес:**
- **Токен YClients** — блокер для всего блока
- Алерт по абонементам (остаток занятий + срок) → Telegram/MAX
- P&L по проведённым занятиям + доход от сгорания отдельной линией
- Расписание/занятость зала для арендаторов (сайт или бот)
- Бот для тренеров: занятость зала и свой график
- Контент-конвейер: LLM-текст + генерация картинок, кириллица накладывается программно
  (генератор не должен рисовать русские буквы), гейт одобрения, публикация в Telegram/VK

**Капстоун:** клиентский ИИ-бот. RAG по прошлым перепискам = **стиль**, живой API = **факты**
(источники строго разделены). Ограждения: не выдумывать цены и наличие, не брать обязательств,
эскалировать нестандартное. Лесенка: черновик → одобрение человеком → автономия.

## Технические заметки

- Боты Telegram и MAX **не могут написать человеку первыми** — нужен предварительный `/start`.
  Проактивные уведомления клиентам идут через SMS/email; бот силён на входящем потоке.
- Боты MAX — только для верифицированных юрлиц РФ (ИП и физлица не могут).
  MAX предпочтительнее как канал: он в белом списке и работает при отключении интернета.
- Instagram Direct и WhatsApp в РФ проблемны (Meta заблокирована) — для автопубликации
  и ботов ориентироваться на Telegram и VK.
- Свежая Ubuntu не даёт ставить пакеты Python в систему (`externally-managed-environment`) —
  всегда работать через `venv`.
